# Лабораторная работа №8. Непрерывная интеграция с помощью Github Actions

## Цель работы

В рамках данной работы студенты научатся настраивать непрерывную интеграцию с
помощью Github Actions на базе контейнеров Docker.

## Задание

Создать Web приложение на базе PHP, написать юнит-тесты для его компонентов и
настроить непрерывную интеграцию с помощью Github Actions, используя
контейнеры Docker.

## Выполнение работы

### Шаг 1. Подготовка репозитория

Создан репозиторий `containers08` и клонирован локально. В корне созданы
директории `site`, `sql`, `tests` и `.github/workflows`.

Итоговая структура проекта:

```text
containers08
├── .github/
│   └── workflows/
│       └── main.yml
├── site/
│   ├── modules/
│   │   ├── database.php
│   │   └── page.php
│   ├── templates/
│   │   └── index.tpl
│   ├── styles/
│   │   └── style.css
│   ├── config.php
│   └── index.php
├── sql/
│   └── schema.sql
├── tests/
│   ├── testframework.php
│   └── tests.php
├── Dockerfile
└── README.md
```

### Шаг 2. Создание Web-приложения

#### Модуль работы с базой данных ([site/modules/database.php](site/modules/database.php))

Реализован класс `Database`, использующий расширение PDO с драйвером
`pdo_sqlite`. Класс содержит методы:

- `__construct($path)` — подключение к SQLite-базе по указанному пути;
- `Execute($sql)` — выполнение произвольного SQL-запроса;
- `Fetch($sql)` — выполнение выборки и возврат результата массивом;
- `Create($table, $data)` — вставка записи из ассоциативного массива;
- `Read($table, $id)` — чтение записи по первичному ключу;
- `Update($table, $id, $data)` — обновление записи;
- `Delete($table, $id)` — удаление записи;
- `Count($table)` — подсчёт количества записей.

Использование подготовленных выражений (`prepare`/`bindValue`) защищает от
SQL-инъекций.

#### Модуль страницы ([site/modules/page.php](site/modules/page.php))

Класс `Page` загружает шаблон через `file_get_contents` и заменяет в нём
плейсхолдеры вида `{{key}}` на значения из переданного ассоциативного массива.

#### Шаблон и стили

Шаблон [site/templates/index.tpl](site/templates/index.tpl) содержит разметку
страницы с плейсхолдерами `{{title}}` и `{{content}}`. Стили вынесены в
[site/styles/style.css](site/styles/style.css).

#### Точка входа и конфигурация

Файл [site/config.php](site/config.php) задаёт путь к SQLite-файлу
(`/var/www/db/db.sqlite`), а [site/index.php](site/index.php) читает запись
`page` по идентификатору из параметра `?page=` и рендерит её через шаблон.

### Шаг 3. Схема базы данных

В файле [sql/schema.sql](sql/schema.sql) описана таблица `page` и добавлены
три тестовые записи. Схема применяется на этапе сборки образа.

### Шаг 4. Тесты

В [tests/testframework.php](tests/testframework.php) размещён минимальный
фреймворк тестирования (функции `info`, `error`, `assertExpression` и класс
`TestFramework`).

В [tests/tests.php](tests/tests.php) реализованы десять тестов:

1. Подключение к базе данных.
2. Метод `Count`.
3. Метод `Create`.
4. Метод `Read`.
5. Метод `Update`.
6. Метод `Delete`.
7. Метод `Execute`.
8. Метод `Fetch`.
9. Рендеринг страницы (`Page::Render`) на временном шаблоне.
10. Рендеринг страницы на реальном шаблоне `index.tpl`.

Каждый тест изолирован в отдельной функции, ошибки перехватываются через
`try/catch`, итог выводится в формате `N / M`.

### Шаг 5. Dockerfile

В [Dockerfile](Dockerfile) используется базовый образ `php:7.4-fpm`,
устанавливаются `sqlite3`/`libsqlite3-dev` и расширение `pdo_sqlite`. На этапе
сборки из `sql/schema.sql` создаётся файл базы `/var/www/db/db.sqlite`, после
чего исходный SQL удаляется. Каталог `/var/www/db` объявлен как `VOLUME`,
поэтому база сохраняется в отдельный том при запуске контейнера.

### Шаг 6. Github Actions

В [.github/workflows/main.yml](.github/workflows/main.yml) настроен workflow
`CI`, запускающийся на `push` в ветку `main`. Шаги:

1. `actions/checkout@v4` — получение исходников.
2. Сборка образа `containers08`.
3. Создание контейнера `container` с подключённым томом `database`.
4. Копирование каталога `tests` внутрь контейнера в `/var/www/html`.
5. Старт контейнера.
6. Запуск тестов: `php /var/www/html/tests/tests.php`.
7. Остановка и удаление контейнера.

### Шаг 7. Запуск

После `git push` workflow запускается автоматически. Результаты видны во
вкладке Actions репозитория; все десять тестов проходят успешно
(`10 / 10`).

## Ответы на вопросы

### Что такое непрерывная интеграция?

Непрерывная интеграция (Continuous Integration, CI) — это практика
разработки, при которой участники команды регулярно (несколько раз в день)
отправляют изменения в общий репозиторий, после чего автоматизированная
система выполняет сборку проекта и прогон тестов. Такой подход позволяет
быстро обнаруживать конфликты и регрессии, обеспечивает воспроизводимость
сборки и сокращает стоимость исправления ошибок, так как они выявляются сразу
после появления.

### Для чего нужны юнит-тесты? Как часто их нужно запускать?

Юнит-тесты проверяют корректность работы отдельных модулей (функций,
классов) в изоляции. Они фиксируют ожидаемое поведение кода, защищают от
регрессий при рефакторинге и служат живой документацией. Запускать их нужно
как можно чаще: локально — при каждом сохранении/сборке, а в CI — при каждом
`push` и при создании Pull Request, чтобы гарантировать, что в основную ветку
попадает только рабочий код.

### Что нужно изменить в файле `.github/workflows/main.yml`, чтобы тесты запускались при каждом создании Pull Request?

Нужно добавить триггер `pull_request` в секцию `on`:

```yaml
on:
  push:
    branches:
      - main
  pull_request:
    branches:
      - main
```

После этого workflow будет запускаться и при `push` в `main`, и при открытии
или обновлении Pull Request, нацеленного на `main`.

### Что нужно добавить в файл `.github/workflows/main.yml`, чтобы удалять созданные образы после выполнения тестов?

Необходимо добавить завершающий шаг, удаляющий созданный образ. Чтобы очистка
выполнялась даже при падении тестов, полезно указать `if: always()`:

```yaml
      - name: Remove the Docker image
        if: always()
        run: docker rmi containers08
```

Дополнительно можно выполнять `docker image prune -f` для удаления
промежуточных слоёв.

предварительно инициализированной SQLite-базой. Настроенный workflow Github
Actions автоматически собирает образ, поднимает контейнер и прогоняет тесты
при каждом изменении в ветке `main`. Практика показала, что контейнеризация
вместе с CI даёт повторяемое и изолированное окружение для тестов, а
добавление триггеров `pull_request` и шага очистки образов позволяет легко
расширять pipeline под нужды команды.
